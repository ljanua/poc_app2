Coach match & game performance history

Date: 2026-07-18 Topic: match-game-performance Mode: repo-grounded Focus: games list, lineup/subs, minutes, Game_Performance, S2 Match History Performance

Grounding context

Codebase: Mockup-first coach app (S0–S9) with scripts/serve-mockup.js + Postgres. Match data today is only denormalized player_stats aggregates (total minutes, appearances, recent avg, average/last-match scores) shown on S2 Match Time History / Recent Performance and editable on S5 — not a game ledger.

Gaps: No games, lineup, substitution, or per-game rating tables. Scope docs already promise “minutes, appearances, substitutions” (docs/plan/scope/project-scope-plan.md, brainstorm 2026-07-01) but implementation never landed event-level data.

Patterns to reuse: Assessment history (player_skill_ratings_history + S9 write / S2 read), club-scoped coach auth, team assignments, audit trail, rollup scripts (clip stats reconcile).

External: Sideline apps (Pitch Planner, FairSub, Pitch Time, Coach Meister) optimize live fair playing time. VantageIQ’s wedge is development history + game rating into the player dashboard — not competing on sideline timers first.

Learnings in docs/solutions/ are infra-only (avatar/birth-month); no match-domain learnings.

Topic axes

  • Fixture ledger — upcoming and past games for a team
  • Lineup & substitution capture — starters, sub in/out, by whom
  • Minutes & season rollup — per-game minutes → player season totals
  • Game rating → Game_Performance + S2 — per player per game score on the dashboard
  • Coach UX surface — where entry lives (new screen vs team hub vs live mode)

Ranked ideas

1. Post-match Game Sheet 2. Event-sourced minutes 3. Team fixture board 4. Game rating ≠ S9 Assessment 5. Auto-rollup player_stats 6. Phased MVP 7. Pre-match lineup → played

#1 Post-match Game Sheet → Game_Performance → S2

Axis: Game rating + UX · Confidence 88% · Complexity Medium

After the whistle, coach opens a Game Sheet for a fixture: set starters, log substitution events (player out, player in, minute), enter a quick 0–10 (or 0–100) game rating per participant. Persist rows in game_performance (player × game × minutes × rating × start/sub flags). S2 section Match History Performance lists recent games with minutes + rating (replace or sit beside today’s aggregate-only Match Time History).

Game fixture→ Sheet: lineup + subs + ratings→ game_performance→ S2 Match History Performance

Basis: direct: S2 already has Match Time / Recent Performance slots fed by player_stats only; Assessment history proves event→timeline UI. external: sideline apps prove demand for minutes; product identity is post-match development record.

Rationale: Closest match to the stated ask; reuses dashboard real estate; creates the source of truth aggregates currently fake/manual.

Downsides: Post-match entry is delayed vs live; coach must remember sub times (mitigate with simple minute picker).

#2 Event-sourced minutes from the sub timeline

Axis: Minutes & rollup · Confidence 85% · Complexity Medium

Do not let coaches type minutes as the primary source. Starters begin at minute 0; each sub event ends one player’s on-pitch interval and starts another’s; final whistle closes open intervals. Displayed minutes = sum of intervals. “Subbed by whom” is the event edge (out → in).

Basis: reasoned: manual minutes drift from the sub story the coach just entered. external: Pitch Planner / FairSub treat the clock + sub taps as ground truth.

Rationale: Makes “who started / who was subbed / by whom / minutes” one coherent model instead of four independent fields.

Downsides: Needs clear rules for half-time, red cards, injury without replacement, and unknown exact minute (allow approximate minute).

#3 Team fixture board (upcoming + past)

Axis: Fixture ledger · Confidence 82% · Complexity Medium

Team-scoped list of games (date, opponent, home/away, status: upcoming / in progress / complete). Entry point from S3 Team Management or a new “Games” nav item. Upcoming rows support pre-lineup; complete rows open the Game Sheet. ClubAdmin sees club-wide fixtures.

Basis: direct: teams + player_team_assignments already anchor roster; games naturally hang off team_id. Scope plan lists match participation history as in-scope.

Rationale: Coaches need a calendar of work, not only player-centric history; without fixtures, Game_Performance has nothing to attach to.

Downsides: Scheduling UX can sprawl (reschedule, cancel, multi-team days) — keep v1 fields minimal.

#4 Keep game rating distinct from S9 skill Assessment

Axis: Game rating · Confidence 90% · Complexity Low

Game rating = holistic match performance (one number per player per game). S9 Assessment = skill-dimension ratings with history. Both can appear on S2 (Assessment History vs Match History Performance) but must not share write paths or overwrite each other.

Basis: direct: S9 + player_skill_ratings_history just shipped; conflating would blur two different coach intents. reasoned: a 7/10 game score is not “Passing = 72”.

Rationale: Prevents schema and UX collision; lets Match History Performance stay scannable.

Downsides: Two rating concepts for coaches to learn — mitigate with clear labels (“Game rating” vs “Skill assessment”).

#5 Auto-rollup player_stats from Game_Performance

Axis: Minutes & rollup · Confidence 80% · Complexity Medium

When a game sheet is saved, recompute season total_minutes, appearances, recent_avg, and optionally average game rating / last match fields on player_stats. Retire S5 as the primary writer for match minutes (display-only or remove), mirroring how Assessment became the skill write path.

Basis: direct: fabricated stats bug history and “source of record” plans already treat player_stats as fragile when not event-backed. Clip reconcile scripts show the rollup pattern.

Rationale: Makes existing S2 aggregate cards truthful; closes the loop from sheet → dashboard without double entry.

Downsides: Migration for seeded demo players; need idempotent recompute and clear “incomplete sheet” behavior.

#6 Phased MVP: participation + minutes + rating before full sub graph

Axis: UX surface · Confidence 78% · Complexity Low→Medium

Ship v1 as: create game → mark who played → enter minutes (or start/end) + game rating → Game_Performance → S2 list. v1.1 adds starters flag and ordered sub events with “replaced by”. Schema can still reserve event tables empty in v1.

Basis: reasoned: full sub graph is the hardest UX; user value of S2 history + ratings arrives earlier. Aligns with Assessment feature’s “partial OK” pragmatism.

Rationale: De-risks delivery while still creating game_performance and the S2 section the user named.

Downsides: Temporary gap vs “by whom” ask — must be an explicit phase, not a silent cut.

#7 Pre-match lineup draft that becomes the played sheet

Axis: Fixture ledger + lineup · Confidence 75% · Complexity Medium

For upcoming games, coach saves a planned XI + bench. At kickoff/status=complete, that draft seeds the Game Sheet (starters pre-checked). Subs during/after edit the same record. Reduces blank-page friction after the match.

Basis: external: Pitch Planner lineup builder → play flow. direct: team roster already known via assignments.

Rationale: Connects “upcoming games” management to the post-match record without two disconnected tools.

Downsides: Planned ≠ actual; need clear “reset from plan” vs “edit after play” states.

Top pick
